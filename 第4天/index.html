    第4天 C语言与画面显示的练习

    用C语言实现内存写入

    想要画东西的话，只要往VRAN里写点什么就可以了。但是在C语言中又没有直接写入指定内存地址的语句。所以，我们干脆就创建一个有这种功能的函数。

    naskfunc.nas
    [FORMAT "WCOFF"]
[INSTRSET "i486p"]         ; 这个程序是给486CPU用的
[BITS 32]
[FILE "naskfun.nas"]

    GLOBAL _io_hlt,_write_mem8

[section .text]

_io_hlt:
    HLT
    RET

_write_mem8:
    MOV    ECX,[ESP+4]     ; [ESP+4]中存放的是地址，将其读入ECX
    MOV    AL,[ESP+8]      ; [ESP+8]中存放的是数据，将其读入ALL
    MOV    [ECX],AL
    RET

    数字的存放地址：[ESP+4], [ESP+8], [ESP+12]...

    如果与C语言联合使用的话，能自由使用的寄存器只有EAX, ECX，EDX。其他寄存器只能使用其值，不能改变其值

        write_mem8(i,15);

    {
        io_hlt();   
    }
}

    挑战指针

    很明显，write_mem8可以用指针代替😍。

    write_mem8(i, i & 0x0f);

    替代以上语句的是：

    i = i & 0x0f;

    但是，会出错：

    invalid type argument of 'unary *'

    我们从编译器的角度稍微想想就能明白为什么会出错了。

    MOV    [0x1234],0x56

    这种情况也会出错。这是因为指定内存时，不知道到底是BYTE，还是WORD，还是DWORD。只有在另一方也是寄存器的时候才能省略，其他情况都不能省略。

    这次，我们费劲写的C语言语句，相当于下面的汇编语句：

    MOV    [i],(i & 0x0f)

    但是却不知道[i]到底是BYTE, 还是WORD, 还是DWORD。

    所以，我们使用：

    char *p;          //用于存放BYTE类地址
p = i & 0x0f;

    同理：

    char *p;       //用于BYTE类地址，相当于AL
short *p;      //用于WORD类地址，相当于AX
int *p;        //用于DWORD类地址，相当于EAX

    但是无论什么情况，变量p都是4字节。这是因为变量p是用于记录地址的变量。在汇编语言中，地址也像ECX一样，用4字节的寄存器来指定，所以也是4字节。

    bootpack.c
    void io_hlt(void);

void HariMain(void)
{
    int i;
    char *p;

    for (i = 0xa0000; i <= 0xaffff; i++)
    {
        p = i;
        *p = i & 0x0f;  
    }

    for (;;)
    {
        io_hlt();   
    }
}

    类型转换

    make run之后，且慢！😮发现有一行警告：

    warning: assignment makes pointer from integer without a cast

    在C语言中，普通数值和表示内存地址的数值被认为是两种不同的东西，虽然事实上没什么不同。

    基于这种设计思想，如果将普通整数值赋给内存地址变量，就会有警告。

    为了避免这种情况发生，可以这样写：

    p = (char*) i;

    使用类型转换就可以不用指针

    p = (char*) i;
*p = i & 0x0f;

    这样就得到下式：

    *((char*)i) = i & 0x0f;

    这种写法就与汇编语言相像：

    MOV    BYTE [i], i & 0x0f

    下面再稍微深入说明一下，我们常见的两个语句是：

    p = (char*) i;
*p = i & 0x0f

    将上面的语句按汇编的习惯写一下：

    MOV    ECX,i
MOV    [ECX],(i & 0x0f)

    它们的区别很清楚，即一个是给ECX寄存器赋值，一个是给[ECX]号内存地址赋值。

    如果执行顺序调过来：

    *p = i & 0x0f;
p = (char*) i;

    这会导致i & 0x0f的结果写入内存的某个不可知的地址中。

    其实，*p并不是什么变量，变量只有p。所谓*p，就相当于BYTE [p]这种语句的代替。

    *的位置

    一般的写法是：

    char *p;

    虽然我们声明的是变量p，写成这样也对：

    char* p;

    但是

    char* p, q;

    q会被看作是一般1字节的变量，为了避免这样的误解，程序员都写成：

    char *p, *q;

    不必看作指针

    char *p;不必看作指针。p不是指针，而是地址变量，这种说法比较好，将地址值赋给地址变量是理所当然的。

    指针的应用(1)

    绘制条纹图案的部分，也可以写成以下这样：

    void io_hlt(void);

void HariMain(void)
{

    char *p;

    p = (char *) 0xa0000;

    for (i = 0; i <= 0xffff; i++)
    {
        *(p + 1) = i & 0x0f;    
    }

    色号设定

    这次使用的是320x200的8位颜色模式，也就是只能使用0~255的数。熟悉电脑颜色的人都知道，这是非常少的。一般说起指定颜色，都是用#ffffff一类的数，这就是RGB方式。

    如果像现在这样，程序员不做任何设定，0号颜色就是#000000，15号颜色就是#ffffff。

    要想描绘一个操作系统摸样的画面，只要有以下这16种颜色就够了：

{
    static unsigned char table_rgb[16 * 3] = {

    0x00, 0xff, 0x00  //2:亮绿
    0xff, 0xff, 0x00  //3:亮黄
    0x00, 0x00, 0xff  //4:亮蓝
    0xff, 0x00, 0xff  //5:亮紫
    0x00, 0xff, 0xff  //6:浅亮蓝
    0xff, 0xff, 0xff  //7:白
    0xc6, 0xc6, 0xc6  //8:亮灰
    0x84, 0x00, 0x00  //9:暗红
    0x00, 0x84, 0x00  //10:暗绿
    0x84, 0x84, 0x00  //11:暗黄
    0x00, 0x00, 0x84  //12:暗青
    0x84, 0x00, 0x84  //13:暗紫
    0x00, 0x84, 0x84  //14:浅暗蓝
    0x84, 0x84, 0x84  //15:暗灰
    };
    set_palette(0, 15, table_rgb);
    return;
    //C语言中的static char语句只能用于数据，相当于汇编中的DB指令。
}

void set_palette(int start, int end, unsigned char *rgb)
{
    int i, eflags;
    eflags = io_load_eflags();  //记录终端许可标志的值
    io_cli(); //将中断许可标志置为0，禁止中断
    io_out8(0x03c8, start);
    for (i = start, i <= end; i++)
    {
        io_out8(0x03c9, rgb[0] / 4);
        io_out8(0x03c9, rgb[1] / 4);
        io_out8(0x03c9, rgb[2] / 4);
        rgb += 3;   
    }
    io_store_eflags(eflags); //复原中断许可标志
    return;
}

    char a[3];

    C语言中，如果这样写，那么a就成为了常数，以汇编的语言讲就是标识符。标识符的值当然就意味着地址。

    上面的叙述就相当于汇编里的这个语句：

    a:
    RESB    3

    nask中的RESB的内容能够保证是0，但C语言中不能保证，所以里边说不定含有某种垃圾数。

    就像在汇编语言中用DB指令代替RESB指令那样，在C语言中也有类似的指示方法，那就是在声明前加上static。

    usigned的意思是：这里所处理的数据是BYTE型，但它是没有符号的数(0或正整数)。

    char型的变量有3种模式，分别是signed型、unsigned型和未指定型。

    对于char来说，signed型能够处理-128~127的整数，unsigned型能够处理0~255的整数。未指定型可由编译器决定是unsigned还是signed。

    在这个程序里，多次出现0xff，也就是255，如果它被误解为负数就很麻烦。

    最后的函数set_palette，下面是精简版：

    CPU与设备相连

    CPU的管脚不仅与内存相连，还与各个设备相连，如声卡、显卡、软盘等等。

    CPU向设备发送电信号是OUT指令；从设备取得电信号的是IN指令。

    在OUT指令和IN指令中，为了区别不同的设备，也要使用设备号码。设备号码在英文中称为port(端口)。

    在C语言中，没有与IN和OUT指令相当的语句，所以我们只好拿汇编语言来做了🤔。

    设备号码0x03c8和0x03c9，可以在osdev中查询VGA：

    Port 0x3c8, 0x3c9, 0x3c7 control the DAC. Each register in the DAC consists of 18 bits, 6 bits for each color component. To write a color, write the color index to port 0x3c8, then write 3 bytes to 0x3c9 in the order red, green, blue. If you want to write multiple consecutive DAC entries, you only need to write the first entry's index to 0x3c8 then write all values to 0x3c9 in the order of red, green, blue, red, green, blue, and so on. The accessed DAC entry will automatically increment after every 3 bytes written. To read the DAC entries, write the index to be read to 0x3c7, then read the bytes from port 0x3c9 in a similar fashion (as with writing, the index will increment after every three bytes read).

    CLI和STI

    CLI将中断标志置为0，STI将中断标志置为1。当CPU遇到中断请求时，是立即处理中断请求(中断标志为1)，还是忽略中断请求(中断标志为0)，就由这个中断标志位来设定。

    EFLAGS寄存器

    这是由名为FLAGS的16位寄存器扩展而来的32位寄存器。FLAGS是存储进位标志和中断标志等标志的寄存器。进位标志可以通过JC或者JNC等跳转指令来简单地判断到底是0还是1。

    但对于中断标志，没有类似的JI或JNI命令，所以只能读入EFLAGS，再检查第9位是0还是1。

    随便说一下，进位标志是EFLAGS的第0位。

    恢复中断标志

    set_palette中想要做的事情是在设定调色板之前首先执行CLI，但处理结束以后一定要恢复中断标志，因此需要记住最开始的中断标志是什么。

    所以我们制作了一个函数io_load_eflags，读取最初的eflags值。

    naskfunc.nas
    [FORMAT "WCOFF"]
[INSTRSET "i486p"]
[BITS 32]
[FILE "naskfunc.nas"]

    GLOBAL _io_hlt, _io_cli, _io_stihlt
    GLOBAL _io_in8, _io_in16, _io_in32
    GLOBAL _io_out8, _io_out16, _io_out32
    GLOBAL _io_load_eflags, _io_store_eflags

[SECTION .text]

_io_hlt:       ; void io_hlt(void);
    HLT
    RET

_io_cli:       ; void io_cli(void);
    CLI
    RET

_io_sti:       ; void io_sti(void);
    STI
    RET

_io_stihlt:    ; void io_stihlt(void);
    STI
    HLT
    RET

_io_in8:       ; void io_in8(int port);
    MOV    EDX,[ESP+4]   ; port
    MOV    EAX,0
    IN     AL,DX
    RET

_io_in16:      ; int io_in16(int port);
    MOV    EDX,[ESP+4]   ; port
    MOV    EAX,0
    IN     AX,DX
    RET

_io_in32:      ; int io_in32(int port);
    MOV    EDX,[ESP+4]   ; port
    IN     EAX,DX
    RET

_io_out8:      ; void io_out8(int port, int data);
    MOV    EDX,[ESP+4]   ; port
    MOV    AL,[ESP+8]    ; data
    OUT    DX,AL
    RET

_io_out16:    ; void io_out16(int port, int data);
    MOV    EDX,[ESP+4]   ; port
    MOV    EAX,[ESP+8]   ; data
    OUT    DX,AX
    RET

_io_out32:    ; void io_out32(int port, int data);
    MOV    EDX,[ESP+4]   ; port
    MOV    EAX,[ESP+8]   ; data
    OUT    DX,EAX
    RET

_io_load_eflags:   ; int io_load_eflags(void);
    PUSHFD         ; 指PUSH EFLAGS
    POP    EAX
    RET

_io_store_eflags:  ; void io_store_eflags(int eflags);
    MOV    EAX,[ESP+4]
    PUSH   EAX
    POPFD          ; 指POP FLAGS
    RET

    如果有MOV EAX,EFLAGS之类的指令就简单多了，但CPU没有这种指令。能够用来读写EFLAGS的，只有PUSHFD和POPFD。

    PUSHFD是“push flags double-word”，意思是将标志位的值按双字长压入栈。其实它能做的，无非就是“PUSH EFLAGS”。

    POPFD是“pop flags double-word”的缩写，意思是按双字节长将标志位从栈弹出。它所作的，就是“POP EFLAGS ”。

    PUSHFD
POP EAX

    代替了：

    MOV    EAX,EFLAGS

    PUSH    EAX
POPFD

    代替了：

    MOV    EFLAGS,EAX

    EAX是返回值

    最后要讲的是io_load_eflags。它对我们而言，是第一个有返回值的函数的例子。

    根据C语言的规约，执行RET语句时，EAX中的值就被看作是函数的返回值。

    绘制矩形

    在当前画面模式中，画面上有320x200 (=64000)个像素。

    假设做商店的坐标是(0,0)，右下点的坐标是(319,319)，那么像素坐标(x,y)对应的VRAM地址计算：

    0xa0000 + x + y*320

    其他画面模式也基本相同，只是0xa0000这个起始地址和y的系数320有些不同。

    根据这种思路我们制作了函数boxfill8。

    bootpack.c
    void io_hlt(void);
void io_cli(void);
void io_out8(int port, int data);
int io_load_eflags(void);
void io_store_eflags(int eflags);

void boxfill8(unsigned char *vram, int xsize, unsigned char c, int x0, int y0, int x1, int y1);

#define COL8_000000 0
#define COL8_FF0000 1
#define COL8_00FF00 2
#define COL8_FFFF00 3
#define COL8_0000FF 4
#define COL8_FF00FF 5
#define COL8_00FFFF 6
#define COL8_FFFFFF 7

#define COL8_008400 10
#define COL8_848400 11
#define COL8_000084 12
#define COL8_840084 13
#define COL8_008484 14
#define COL8_848484 15

void HariMain(void)
{
    char *p;

    init_palette();

    p = (char *) 0xa0000;

    boxfill8(p, 320, COL8_FF0000, 20, 20, 120, 120);
    boxfill8(p, 320, COL8_00FF00, 70, 50, 170, 150);
    boxfill8(p, 320, COL8_0000FF, 120, 80, 220, 180);

    for (;;)
    {
        io_hlt();   
    }
}

void init_palette(void)
{
    static unsigned char table_rgb[16 * 3] = {
        0x00, 0x00, 0x00,
        0xff, 0x00, 0x00,
        0x00, 0xff, 0x00,
        0xff, 0xff, 0x00,
        0x00, 0x00, 0xff,
        0xff, 0x00, 0xff,
        0x00, 0xff, 0xff,
        0xff, 0xff, 0xff,
        0xc6, 0xc6, 0xc6,
        0x84, 0x00, 0x00,
        0x00, 0x84, 0x00,
        0x84, 0x84, 0x00,
        0x00, 0x00, 0x84,
        0x84, 0x00, 0x84,
        0x00, 0x84, 0x84,
        0x84, 0x84, 0x84    
    };
    set_palette(0, 15, table_rgb);
    return;
}

void set_palette(int start, int end, unsigned char *rgb)
{
    int i, eflags;
    eflags = io_load_eflags();
    io_cli();
    io_out8(0x03c8, start);
    for (i = start; i <= end; i++)
    {
        io_out8(0x03c9, rgb[0] / 4);
        io_out8(0x03c9, rgb[1] / 4);    
        io_out8(0x03c9, rgb[2] / 4);
        rgb += 3;
    }
    io_store_eflags(eflags);
    return;
}

void boxfill8(unsigned char *vram, int xsize, unsigned char c, int x0, int y0, int x1, int y1)
{
    int x, y;
    for (y = y0; y <= y1; y++)
    {
        for (x = x0; x <= x1; x++)
        {
            vram[y * xsize + x] = c;    
        }
    }
    return;
}
